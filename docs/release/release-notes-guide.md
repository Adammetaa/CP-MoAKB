# Release Notes Guide

Release notes should identify the reviewed version and commit, describe user-
visible changes, compatibility classification, supported Python versions,
dependency/security review, validation results, migration needs, and known
limitations. Distinguish documentation, internal implementation, public API,
contract, and knowledge changes.

Do not claim production readiness, certification, agricultural efficacy,
publication destinations, or completed deployment unless independently verified.
Do not include secrets, machine paths, usernames, or transient CI identifiers.
