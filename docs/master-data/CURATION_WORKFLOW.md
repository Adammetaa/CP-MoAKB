# Curation Workflow

## Future workflow

```text
Source candidate
→ Source assessment
→ Scope approval
→ Extraction proposal
→ Structured candidate record
→ Provenance attachment
→ Terminology review
→ Domain review
→ Cross-source corroboration
→ Conflict resolution
→ Architecture validation
→ Dataset acceptance
→ Publication
→ Maintenance and re-review
```

No transition is automatic. Rejection, revision, or return to an earlier stage must preserve the reason and reviewer.

## Conceptual responsibilities

| Role | Responsibility |
| --- | --- |
| Source Curator | Registers source identity/version, rights, retrieval, extraction method, and limitations. |
| Data Steward | Controls scope, candidate lifecycle, duplicates, completeness claims, and audit trail. |
| Terminology Reviewer | Reviews Thai/English forms, definitions, synonyms, script, and ambiguity. |
| Domain Reviewer | Reviews agricultural meaning and appropriate contextual relationships. |
| Taxonomy Reviewer | Verifies scientific identity, rank, authority mapping, and taxonomic uncertainty. |
| Evidence Reviewer | Evaluates claim-level support, contradiction, quality, and inference boundaries. |
| Regulatory Reviewer | Separates and validates jurisdiction- and effective-date-specific legal statements. |
| Architecture Governor | Checks entity boundaries, provenance, identity, partitioning, and Design Freeze. |
| Release Manager | Produces reproducible release evidence, notes, approvals, and artifacts. |

Roles are governance functions, not claims that these positions are currently staffed. One person may perform multiple functions only when independence and conflict-of-interest requirements are documented.

## Conflict handling

Conflicting sources are retained as competing statements until reviewed. Curators must not resolve conflicts through majority count, source-domain prestige, label similarity, or silent normalization. Resolution records identify the claim, source versions, reasoning, reviewers, uncertainty, and affected release.
