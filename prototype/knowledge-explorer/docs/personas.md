# User Personas

All journeys describe future product intent using fictional placeholder content.

## Farmer

- **Goals:** Understand plain-language concepts and identify which authority or
  source owns a statement.
- **Pain points:** Technical jargon, hidden uncertainty, and interfaces that
  blur education with advice.
- **Expected journey:** Home search → clear concept summary → evidence and
  authority → explicit “not diagnosis or recommendation” boundary.

## Agronomist

- **Goals:** Compare definitions, evidence scope, lifecycle, and related concepts.
- **Pain points:** Untraceable summaries and context-free claims.
- **Expected journey:** Search with facets → concept → evidence chain → source
  applicability → related domain concepts.

## Crop Advisor

- **Goals:** Locate governed reference material and understand jurisdictional and
  temporal limits.
- **Pain points:** Classification presented as recommendation and stale sources.
- **Expected journey:** Browse domain → concept → authority scope → source
  version → boundary notice.

## Researcher

- **Goals:** Reconstruct evidence, provenance, uncertainty, and disagreements.
- **Pain points:** Flattened citations and silent conflict resolution.
- **Expected journey:** Search evidence → evidence record → source locator →
  conflicts and review history → citation export placeholder.

## Student

- **Goals:** Learn how concepts relate and why evidence and authority matter.
- **Pain points:** Dense taxonomies and unexplained governance language.
- **Expected journey:** Featured domain → concept overview → relationship map →
  evidence explainer → glossary cues.

## Government Officer

- **Goals:** Verify official authority, jurisdiction, version, and publication
  status.
- **Pain points:** Scientific context mistaken for legal permission.
- **Expected journey:** Authority directory → authority profile → governed source
  → related claims → explicit regulation boundary.

## Developer

- **Goals:** Understand the platform boundaries and future information model
  without mistaking the prototype for an API.
- **Pain points:** UI mocks that imply unsupported Runtime capabilities.
- **Expected journey:** About → architecture → governance → component library →
  prototype boundary documentation.

## Knowledge Author

- **Goals:** See how authored meaning, evidence, status, and relationships will
  appear to readers.
- **Pain points:** Templates detached from the eventual reading experience.
- **Expected journey:** Governance → KAS/KGS overview → concept anatomy → evidence
  card → lifecycle presentation.

## Knowledge Reviewer

- **Goals:** Inspect evidence, authority, conflict, review state, and publication
  boundaries together.
- **Pain points:** Approval controls separated from the content being judged.
- **Expected journey:** Governance status → concept review context → evidence and
  authority → lifecycle → audit-trail placeholder.
